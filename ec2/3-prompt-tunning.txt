배치 인퍼런스로 학습 데이터를 생성... 이 부분 말이야.. 개발 중에도 동일 영상에 대해서 여러번 발생할까?

개발 중 같은 영상을 재인퍼런스하게 되는 지점들
앞서 정리하신 챕터 3(프롬프트 구체화/튜닝)이 바로 그 반복의 핵심입니다.

프롬프트 튜닝: 프롬프트를 한 번 고칠 때마다 같은 샘플 영상에 다시 돌려서 결과를 비교합니다. 이게 수십 번 반복돼요.
출력 스키마 변경: JSON 필드를 추가/수정하면 같은 영상을 다시 돌려야 함
프레임 샘플링 파라미터 조정: fps, 프레임 수, 해상도를 바꾸면 입력이 달라지니 재인퍼런스
모델/디코딩 설정 변경: temperature, max_tokens 등 바꿀 때
디버깅: 이상한 출력이 나온 특정 영상만 반복해서 돌려봄
즉 개발 단계에서는 "같은 영상 × 여러 설정 조합"으로 인퍼런스가 중복 발생하는 게 당연합니다.

그래서 설계에 반영할 점
78B 모델 인퍼런스는 비싸고 느리니까, 이 반복을 그냥 두면 개발 사이클이 느려지고 GPU 비용이 낭비돼요. 몇 가지 대응을 권합니다.

1. 개발용 소규모 서브셋 고정 전체 43,000개가 아니라 대표 영상 20~50개만 뽑아 개발 세트로 고정하세요. 카테고리·길이·대화유무 등이 다양하게 섞이도록. 프롬프트 튜닝은 이 작은 세트에서만 돕니다.

2. 캐싱 / 멱등성(idempotency) 설계 결과를 캐시하되, 캐시 키에 "무엇이 바뀌면 재인퍼런스해야 하는지"를 반영하는 게 핵심입니다.


캐시 키 = hash(영상ID + 프레임샘플링설정 + 프롬프트버전 + 모델버전 + 디코딩설정)
프롬프트만 바꿨는데 프레임은 그대로면 → 프레임 추출은 캐시 재사용, 인퍼런스만 다시
아무것도 안 바뀌면 → 인퍼런스도 스킵
이렇게 하면 "바뀐 것만 다시 계산"이 됩니다.

3. 단계 분리로 재계산 최소화 파이프라인이 영상 → 프레임 샘플링 → 인퍼런스인데, 프레임 샘플링 결과(이미지)를 S3에 저장해두면 프롬프트만 튜닝할 때 프레임 추출을 매번 다시 안 해도 됩니다. Graviton(CPU)에서 하는 프레임 샘플링과 GPU 인퍼런스를 분리해둔 구조가 이미 이걸 잘 받쳐줘요.

4. 프롬프트/출력 버전 태깅 생성한 JSON 라벨에 prompt_version, model_version, sampling_config를 메타로 박아두면, 나중에 "이 라벨이 어떤 설정으로 만들어졌나"를 추적할 수 있고 중복/재현 관리가 쉬워집니다.

프로덕션(EKS)에서는?
반대로 프로덕션 배치 인퍼런스에서는 같은 영상을 두 번 돌리면 안 됩니다 (비용 낭비). 그래서 EKS 단계에선:

이미 처리된 영상은 건너뛰는 체크포인트/멱등 처리 (S3에 결과 있으면 스킵)
K8s Job이 중간에 죽어도 재실행 시 처리 완료분은 건너뛰도록 설계
aws s3 sync가 이미 올라간 파일 건너뛰는 것과 같은 원리예요.

정리하면: 개발 중에는 같은 영상 재인퍼런스가 반복적으로 발생하는 게 정상이고, 이걸 전제로 (a) 소규모 개발 세트 고정, (b) 설정을 반영한 캐싱, (c) 프레임 추출과 인퍼런스 분리를 설계해두면 개발 속도와 비용을 둘 다 잡을 수 있습니다.

혹시 이 캐싱/멱등성 부분을 워크샵 문서에 한 단락으로 추가하고 싶으시면, 챕터 3(프롬프트 튜닝)이나 도입부에 들어갈 문장으로 정리해 드릴게요. 아니면 실제 캐싱 로직(해시 키 생성)을 코드로 잡아드릴 수도 있어요.
